## Setup

- Cleaned stale DB and encryption key files before boot; API started cleanly (PID 38529).
- UI dev server `pnpm run dev` wraps vite with `portless` which failed because `ui.swarm.localhost` was already registered by another PID. Worked around by running `pnpm exec vite --port 5274 --host 127.0.0.1` directly. **Non-blocking but worth a note** — anyone running this QA in parallel with other swarm worktrees will hit this.
- Browser driven via `qa-use browser create --tunnel` (session `f9002728-3ea6-496b-9d1f-161aafe82977`), kept alive as a nohup bg process so the tunnel/browser context persisted across commands.
- Seeded two entries via `PUT /api/config`:
  - `OPENAI_API_KEY` = `sk-live-abc123` (isSecret=true)
  - `MODEL_NAME` = `gpt-4` (isSecret=false)
- Seed API responses confirm the new `encrypted: true|false` field is returned by the HTTP layer and the secret value is masked as `********` in the API response.
- Raw DB check after seed: `OPENAI_API_KEY` value stored as `h7OZ7Sv2pj/a4bqJfT7H6m8HslPriLp3PkSZpZJQ...` (base64 ciphertext, `encrypted=1`). `MODEL_NAME` stored as plaintext `gpt-4` with `encrypted=0`. Encryption at rest confirmed.

### 6. Reserved-key rejection (`API_KEY`)
- **Status:** pass (backend) / **UX gap** (frontend)
- **Screenshot:** /tmp/qa-pr336-screens/05-reserved-key-error.png
- **Notes:**
  - Backend correctly rejects both `API_KEY` and (case-insensitively) `secrets_encryption_key` with HTTP 400 and the exact spec'd message: `Key 'API_KEY' is reserved and cannot be stored in swarm_config. Set it as an environment variable instead.` Verified via direct `curl`.
  - **The UI silently swallows the 400.** The Add Config Entry dialog closed as if the operation succeeded. No toast, no inline error, no dialog-retention. The row does not appear in the grid (correct — nothing was persisted), but a user trying to add `API_KEY` would get zero feedback and would likely think the operation succeeded until they noticed the row is missing.
  - Console confirms the 400 hit the frontend: `[ERROR] Failed to load resource: the server responded with a status of 400 (Bad Request) at http://localhost:5274/config`.
  - Raw DB check confirms `API_KEY` row does NOT exist.
- **This is the single most important finding in this QA run.**

```
ANTHROPIC_API_KEY|eCNj4HwAyuUBEiYSiT+bbAz0K8ztUwq7xIWukbhp|1|1
```

Columns: `key | substr(value,1,40) | encrypted | isSecret`.

- Both isSecret rows have `encrypted=1` and opaque base64 ciphertext — zero plaintext secrets on disk.
- Non-secret row (`telemetry_installation_id`) is plaintext, `encrypted=0`.
- Reserved-key attempt left no residue.

Reserved-key direct API check:

```
PUT /api/config {"key":"API_KEY",...} -> HTTP 400
{"error":"Key 'API_KEY' is reserved and cannot be stored in swarm_config. Set it as an environment variable instead."}

PUT /api/config {"key":"secrets_encryption_key",...} -> HTTP 400
{"error":"Key 'secrets_encryption_key' is reserved and cannot be stored in swarm_config. Set it as an environment variable instead."}
```

Case-insensitive matching confirmed (lowercase form is still rejected).

## UX suggestions (non-blocking, post-merge)

1. **Surface the reserved-key 400 in the UI.** Scenario 6 is the one thing that genuinely needs a follow-up. The `upsertConfig` mutation in `new-ui/src/api/hooks/use-config-api.ts` (or wherever the hook lives) should expose the error to the dialog via a toast or inline error. Currently the dialog closes optimistically on submit regardless of backend success.
2. **Show an `encrypted at rest` indicator.** The new `encrypted` boolean on `SwarmConfig` is not surfaced anywhere in the UI. A small shield/lock icon next to `isSecret` rows in the grid (or a line in the detail dialog: "Stored encrypted at rest") would give operators confidence that the hardening is live. Pure polish — not ship-blocking.
3. **Consider a banner that lists reserved keys** on the Add Config dialog, or inline validate the key field client-side against `["API_KEY","SECRETS_ENCRYPTION_KEY"]` (case-insensitive) before the PUT fires — avoids the round-trip entirely.
4. The `portless` wrapper on `pnpm run dev` collides with other swarm worktrees running in parallel. Unrelated to this PR but bit me during setup.

## Verdict

**ship with caveats** — the encryption-at-rest and reserved-key backend behavior works exactly as spec'd: ciphertext in DB, transparent decrypt on read, correct 400 on reserved keys. The caveat is purely UX: the frontend swallows the reserved-key 400 silently, which will confuse anyone who tries to store `API_KEY` via the dashboard. This should be a follow-up PR (trivial), not a blocker for PR #336.
