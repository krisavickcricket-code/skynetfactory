# PR Auto-Merge Safety Controls — Brainstorm

## Context

Users are reporting that their agent-swarm deployments are auto-merging PRs too aggressively — specifically, PRs generated by swarm agents are getting merged without sufficient human review.

The suspected root cause is the chain of automatic GitHub events we have wired up (e.g., on PR creation, on check completion, on approval). These events can cascade and result in a PR being merged before a human has meaningfully reviewed it.

This is a trust and safety problem: the swarm is designed to augment human developers, not bypass them. If PRs merge before humans can review them, it undermines the core value proposition.

**Key tension:** We want automation (fast feedback loops, CI integration) but we also want human oversight (meaningful review before merge). The current system may be too far on the automation side.

## Exploration

### Q: What does the typical problematic cascade look like?
Event cascade triggers merge. A chain of GitHub events (PR created → checks pass → review submitted) triggers agent actions that culminate in merge without meaningful human review.

**Insights:** The auto-merge isn't hardcoded — it's emergent. The codebase handles `pull_request` (opened, edited, synchronize, closed), `pull_request_review` (submitted), `check_run`, `check_suite`, and `workflow_run` events. Each event can create a task for an agent, and the agent's response to that task may include GitHub actions (approve, merge). The problem is the speed and automaticity of this chain — events fire faster than humans can intervene.

### Q: Are all agent-created PRs treated the same by the event system?
All PRs treated the same. No distinction — every agent-created PR goes through the same event cascade regardless of risk/complexity.

**Insights:** This is a major lever. If all PRs are equal in the eyes of the system, a one-line typo fix and a 500-line refactor both get the same treatment. Without risk categorization, there's no way for the system to know when human review is critical vs when auto-merge is acceptable. This is likely where the biggest improvement opportunity lives.

### Q: What's the typical user setup, and do they have branch protection?
Mix of multi-agent setups: some with minimal branch protection (agents merge freely), some with branch protection that agents satisfy (e.g., agent approval counts as a valid review). Users had to add branch protection reactively to "fix" the auto-merge problem.

**Insights:** This confirms the problem is in our defaults, not user misconfiguration. Users shouldn't have to layer GitHub branch protection on top as a band-aid. The swarm itself should have sane defaults for merge safety. Also notable: when agents can satisfy branch protection requirements (e.g., counting as an approved reviewer), protection rules don't actually protect against agent-initiated merges.

### Q: What should the desired end state look like?
A combination: (1) never auto-merge by default, (2) configurable merge policy for users who want automation, and (3) human-in-the-loop at the review stage — agent approvals shouldn't count as sufficient for merge.

**Insights:** This is a layered approach:
- **Layer 1 — Safe default:** No auto-merge out of the box. This protects new users immediately.
- **Layer 2 — Policy engine:** Users can opt-in to auto-merge with configurable rules (file count, diff size, file types, labels, etc.).
- **Layer 3 — Review gate:** Even when auto-merge is enabled, require at least one human reviewer's approval (agent reviews don't count toward merge eligibility).

This means the system needs to understand the concept of "human vs agent" at the review level, not just at the PR creation level.

### Q: Where should the merge policy configuration live?
Swarm config (API-side), so we can have multi-level rules. Centrally managed through the existing config system.

**Insights:** This is a good call. The swarm config system already supports scoping (global, agent-level). We could extend this with repo-level scoping too. Multi-level rules means: global defaults (safe) → repo-level overrides → maybe agent-level overrides. This avoids file-in-repo sprawl and keeps policy management centralized, which is important for teams where a single person administers the swarm across many repos. The existing `set-config` / `get-config` tools would be the management interface.

### Q: Should we control the events agents see, the actions they can take, or both?
Control the events — filter/suppress. Don't forward certain event combinations to agents so they don't get triggered to merge in the first place.

**Insights:** This is the "don't put the temptation in front of them" approach. If an agent never receives a "checks passed on your PR" event, it can't decide to merge. This is simpler and more reliable than action-level gating (which requires intercepting merge API calls and checking policy — more complex, more failure modes).

However, there's a nuance: event filtering alone might not be enough if agents can independently decide to check PR status and merge. We should consider whether agents have the *tools* to merge (GitHub MCP tools, `gh` CLI access) independent of events. But as a primary control point, event filtering is clean.

### Q: Which events should trigger agent action, and how should the rest be handled?
Only trigger on clear, explicit human actions: assigned as reviewer, tagged/mentioned, etc. — not on automated cascade events (CI passed, checks completed). But make this configurable, with routing logic. Ideally, events become triggers for the **workflow system** — e.g., "trigger workflow on event: pr_created".

**Insights:** This is a powerful architectural shift. Instead of the current model (GitHub event → hardcoded handler → create task), the model becomes:

```
GitHub event → event bus → workflow trigger matching → workflow execution
```

This means:
1. **Default safety:** Out of the box, only explicit human actions (assign reviewer, @mention, label) trigger agent work.
2. **Configurable automation:** Users who want more automation define workflows with event triggers. They opt-in to each event type explicitly.
3. **Composability:** The workflow system already exists. Events become first-class workflow triggers, and users get the full power of workflow conditions, steps, and approval gates.
4. **Separation of concerns:** The GitHub webhook handler becomes a thin event emitter. Policy and routing live in the workflow layer.

This is arguably a better solution than a "merge policy" config — it's a general-purpose event routing system that subsumes merge policy as one use case.

### Q: Should we ship a quick fix first or go straight to the workflow routing?
Quick win + design doc for workflow. Ship a conservative default event filter ASAP, then write up the workflow routing design for the longer-term solution.

**Insights:** Good pragmatic approach. Two workstreams:
1. **Quick win (days):** Change the default event handling to only create tasks for explicit human actions. Other events either suppressed or logged. This is a config change + small code change in the GitHub webhook handlers.
2. **Design doc (concurrent):** Formalize the "events as workflow triggers" architecture. This feeds into a `/create-plan` for the workflow routing feature.

### Q: Which GitHub events should still trigger agent tasks by default (quick win)?
@mention + assign + label. Only: agent @mentioned in PR/issue, assigned as reviewer, or specific label added (e.g., `swarm-review`).

**Insights:** This is the right balance. The events that should still trigger:
- `issue_comment` / `pull_request_review_comment` — where agent is @mentioned
- `pull_request` with `review_requested` action — agent assigned as reviewer
- `issues` with `labeled` action — specific swarm labels
- `pull_request` with `labeled` action — specific swarm labels

Events that get suppressed by default:
- `check_run` / `check_suite` / `workflow_run` — CI status events
- `pull_request` with `opened` / `synchronize` — PR creation/update (unless agent is mentioned/assigned)
- `pull_request_review` with `submitted` — review approvals from other agents
- `push` events

This means: if nobody explicitly asks an agent to look at a PR, the agent won't look at it. No more cascade.

### Q: What about PRs that agents create themselves — should they notify the lead?
No extra notification needed. There's already a completion notification task when an agent finishes its work (which includes PR creation). That's sufficient. The GitHub `PR opened` event from agent-created PRs should be suppressed like any other automated event.

**Insights:** This simplifies the implementation. We don't need special handling for "agent-created vs human-created" PRs in the event filter. We just filter all events the same way: only explicit human actions (@mention, assign, label) pass through. The existing task completion flow handles the "lead knows about the PR" need.

### Q: How should power users opt in to more aggressive automation?
Workflow-only. No config toggle — if users want automation beyond the safe defaults, they build a workflow with event triggers. The event routing IS the workflow system.

**Insights:** This is the cleanest design. Instead of building a config system that toggles event forwarding, we make the quick win the permanent default: only explicit human actions trigger agents. For anything beyond that, users define workflows. This means:
- No "merge policy" config to build and maintain
- No preset profiles to document and support
- The workflow system is the single extension point for automation
- Users who want "CI passed → auto-merge" build a workflow for it, with whatever conditions and gates they want

This also aligns the quick win with the long-term vision: the quick win restricts default events, and the workflow trigger feature is the only way to expand them. No intermediate toggle system that gets deprecated later.

## Synthesis

### Key Decisions

1. **Safe defaults over configuration.** The default behavior changes: only explicit human actions (@mention, reviewer assignment, specific labels) trigger agent tasks from GitHub events. All other events (CI status, PR opened, review submitted, push) are suppressed by default.

2. **No intermediate toggle/policy system.** We're not building a "merge policy" config, preset profiles, or per-event toggles. The quick win becomes the permanent default. Extension is through the workflow system only.

3. **Event routing via workflows (long-term).** GitHub events become first-class workflow triggers. Users who want automation beyond safe defaults build workflows (e.g., "on check_suite completed + all reviews approved → assign merge task to lead"). This is the single extension mechanism.

4. **Swarm config, not repo files.** Policy lives in the swarm's config system (API-side), not in per-repo config files. This enables multi-level rules (global → repo → agent) through the existing config infrastructure.

5. **Agent-created PRs get no special handling.** They follow the same event filtering as any PR. The existing task completion notification already informs the lead about PR creation.

### Open Questions (Resolved)

- **Event taxonomy for workflows:** ~~New schema needed?~~ **Resolved:** Existing infrastructure already supports this. `WorkflowEventBus` (event-bus.ts) + `handleScheduleTrigger` / `handleWebhookTrigger` (triggers.ts) provide the pattern. GitHub events become a new trigger type alongside `schedule` and `webhook`.
- **Label convention:** Default label(s) configurable by env var (e.g., `GITHUB_EVENT_LABELS=swarm-review,agent-review`). Ships with a sensible default.
- **Backward compatibility:** ~~Migration concern~~ **Resolved:** It’s fine to break. Ship it with a changelog entry explaining the change and why.
- **Event logging:** Not for now. Info-level logging of suppressed events is sufficient.
- **Workflow trigger matching:** Event-based, following the same pattern as schedule triggers. Users define a new trigger type (e.g., `{ type: “github-event”, event: “pull_request”, action: “opened” }`) on their workflows.

### Constraints Identified

- **Quick win must not break the lead's awareness.** The lead needs to know when agents complete tasks (including PR creation). This already works via task completion notifications, independent of GitHub events.
- **GitHub webhook handler is the control point.** The change is in `src/github/` — this is where events are received and tasks/messages are created. The filter goes here.
- **Workflow system maturity.** The workflow system needs to support event triggers before users can opt in to advanced automation. Until then, the quick win is the only behavior.
- **GitHub token permissions.** Some users' GitHub tokens may have merge permissions. The event filter doesn't prevent an agent from merging if it independently decides to — it just removes the trigger. For full safety, consider whether the agent's GitHub token should have reduced permissions by default.

### Core Requirements

**Quick Win (ship ASAP):** ✅ Confirmed — info logging for suppressed events is fine.
1. GitHub webhook handler defaults to only creating tasks/messages for:
   - Agent @mentioned in a comment (issue_comment, pull_request_review_comment)
   - Agent assigned as reviewer (pull_request.review_requested)
   - Specific label(s) added to PR/issue (labeled action)
2. All other GitHub events are silently dropped (or logged at debug level)
3. Existing task completion notifications continue unchanged
4. Document the change and the rationale in changelog

**Workflow Event Triggers (follow-up):**
1. GitHub events are emitted to an internal event bus
2. Workflows can define event triggers (event type + optional conditions)
3. When a matching event fires, the workflow is triggered with event data as `trigger` context
4. Default: no workflows are pre-configured (safe default stands)
5. Documentation with example workflows for common automation patterns (auto-review, auto-merge with gates, etc.)

## Existing Infrastructure (discovered during review)

The codebase already has the building blocks for the long-term solution:

- **`src/workflows/event-bus.ts`** — `WorkflowEventBus` (in-process EventEmitter). Already used by `src/http/webhooks.ts` to emit GitHub events.
- **`src/workflows/triggers.ts`** — `handleScheduleTrigger()` and `handleWebhookTrigger()`. Schedule triggers already find workflows by schedule ID and execute them. The GitHub event trigger would follow the same pattern.
- **`src/http/webhooks.ts`** — Already emits to `workflowEventBus` after processing GitHub events.

The long-term "GitHub event → workflow trigger" feature is a natural extension of the schedule trigger pattern, not a greenfield build.
