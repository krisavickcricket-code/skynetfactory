# Plan: Templates Registry for Agent-Swarm Workers

## Overview

Build a templates registry system at `templates.agent-swarm.dev` — a Next.js static site for browsing pre-configured worker templates, with an interactive docker-compose builder and API routes for worker consumption. Workers use a `TEMPLATE_ID` env var to fetch curated profile content (soul, identity, tools, claude md, startup script) on first boot instead of generic defaults.

## Current State

- Agent profiles have 5 versionable fields: `claudeMd`, `soulMd`, `identityMd`, `toolsMd`, `setupScript` (`src/types.ts:143-177`)
- Generic defaults generated by `generateDefault*()` functions in `src/be/db.ts:2107-2276`
- Default generation triggered at `src/commands/runner.ts:1900-1910` when profile fields are empty
- `docker-compose.example.yml` is a static file with manually duplicated service blocks
- No template, `TEMPLATE_ID`, or docker-compose generation concept exists
- Dashboard (`new-ui/`) is Vite+React (not Next.js) with Space Grotesk/Mono fonts, amber+zinc oklch palette, shadcn/ui new-york style, dark mode default

## Desired End State

1. **`templates/` directory** at repo root with 5 official templates (lead, coder, researcher, reviewer, tester), each containing `config.json` + profile files
2. **`templates-ui/` Next.js app** with:
   - Template gallery page (card grid)
   - Template detail pages with file previews
   - Interactive docker-compose builder
   - API routes serving template JSON for worker consumption
   - Matching branding from `new-ui/`
3. **`runner.ts` integration**: `TEMPLATE_ID` env var → fetch template from registry → use as initial profile (with `{{var}}` interpolation)
4. **Updated `docker-compose.example.yml`** with `TEMPLATE_ID` per service
5. **Local template caching** on workers to avoid repeated network fetches

## Architecture

```
templates/                          # Template data (repo root)
  official/
    lead/config.json + files
    coder/config.json + files
    researcher/config.json + files
    reviewer/config.json + files
    tester/config.json + files
  community/                        # Empty for now

templates-ui/                       # Next.js app
  src/app/
    layout.tsx                      # Root layout + branding
    page.tsx                        # Gallery
    [category]/[name]/page.tsx      # Detail
    builder/page.tsx                # Docker-compose builder
    api/templates/[...slug]/route.ts # JSON API for workers
  src/components/                   # UI components
  src/lib/templates.ts              # FS-based template loading
```

## Constraints

- Templates site is SSG (static export) — no runtime server needed for hosting
- API routes run at build time for static JSON, OR as serverless functions on Vercel
- Template files use `{{agent.name}}`, `{{agent.role}}` etc. — interpolated by worker at boot using existing `interpolate()` from `src/workflows/template.ts`
- The API returns raw template content (with `{{placeholders}}`); the worker interpolates locally
- Versioning: `TEMPLATE_ID=official/coder@1.0.0` or `official/coder` (latest)
- Taras fills real template content later — we create placeholder content

## Phase 6: API Routes for Worker Consumption

Create API routes that serve template JSON for workers to fetch at boot time.

### Files to Create

**`templates-ui/src/app/api/templates/[...slug]/route.ts`**:

Handles:
- `GET /api/templates` — list all templates (configs only, no file content)
- `GET /api/templates/official/coder` — full template with files (latest version)
- `GET /api/templates/official/coder@1.0.0` — specific version (for now, just validates version matches)

Response format — returns **raw content with `{{placeholders}}`** (not interpolated). The worker does interpolation locally with its own agent context. This keeps the API stateless and cacheable.

```json
{
  "config": { "name": "coder", "version": "1.0.0", ... },
  "files": {
    "claudeMd": "# {{agent.name}} - Coder Agent\n...",
    "soulMd": "...",
    "identityMd": "...",
    "toolsMd": "...",
    "setupScript": "#!/bin/bash\n..."
  }
}
```

**Rationale**: Raw templates are simpler to cache (same response for all agents using the same template). The worker already has access to `interpolate()` from `src/workflows/template.ts` and the agent context. The UI detail page shows placeholders highlighted in amber for discoverability.

**Note**: Since `output: "export"` doesn't support API routes, we need to either:
- Option A: Switch to `output: "standalone"` for Vercel deployment (API routes work as serverless functions)
- Option B: Pre-generate JSON files at build time and serve them statically

**Decision**: Use Option A (standalone) — Vercel handles this natively, and we need dynamic version resolution anyway. Update `next.config.ts` to remove `output: "export"`.

### CORS

Add `Access-Control-Allow-Origin: *` headers so workers from any domain can fetch templates.

```typescript
import { interpolate } from "../workflows/template.ts";

async function fetchTemplate(
  templateId: string,
  registryUrl: string,
  cacheDir: string,
): Promise<TemplateResponse | null> {
  // 1. Check local cache: ${cacheDir}/${templateId}.json
  // 2. If cached and not expired (24h), return cached
  // 3. Fetch from ${registryUrl}/api/templates/${templateId}
  // 4. Cache response to ${cacheDir}/${templateId}.json
  // 5. Return response, or null on failure
}
```

Modify the default generation block (lines 1900-1933):

```typescript
if (!agentSoulMd || !agentIdentityMd || !agentToolsMd || !agentClaudeMd) {
  const templateId = process.env.TEMPLATE_ID;
  const registryUrl = process.env.TEMPLATE_REGISTRY_URL || "https://templates.agent-swarm.dev";

  if (templateId) {
    const template = await fetchTemplate(templateId, registryUrl, "/workspace/.template-cache");
    if (template) {
      const ctx = {
        agent: {
          name: agentProfileName || agentName,
          role: role,
          description: agentDescription || "",
          capabilities: (config.capabilities || []).join(", "),
        },
      };
      agentSoulMd = agentSoulMd || interpolate(template.files.soulMd, ctx);
      agentIdentityMd = agentIdentityMd || interpolate(template.files.identityMd, ctx);
      agentToolsMd = agentToolsMd || interpolate(template.files.toolsMd, ctx);
      agentClaudeMd = agentClaudeMd || interpolate(template.files.claudeMd, ctx);
      agentSetupScript = agentSetupScript || interpolate(template.files.setupScript, ctx);

      // Also apply agent defaults from template if not already set
      // (role, capabilities are already set from env/registration)
    }
  }

  // Fallback to generic defaults for any still-missing fields
  const agentInfo = { name: agentProfileName || agentName, role, description: agentDescription, capabilities: config.capabilities };
  agentSoulMd = agentSoulMd || generateDefaultSoulMd(agentInfo);
  agentIdentityMd = agentIdentityMd || generateDefaultIdentityMd(agentInfo);
  agentToolsMd = agentToolsMd || generateDefaultToolsMd(agentInfo);
  agentClaudeMd = agentClaudeMd || generateDefaultClaudeMd(agentInfo);
}
```

## Unit Test: Template Idempotency (CRITICAL)

**File**: `src/tests/template-fetch.test.ts`

A dedicated unit test that verifies templates are NEVER re-applied after the first boot. This is the most important correctness guarantee of the system.

**Test cases**:
1. **First boot with TEMPLATE_ID**: Agent has empty profile fields → template is fetched → profile fields populated with interpolated content → PUT to API saves them
2. **Second boot (restart) with TEMPLATE_ID**: Agent already has profile fields from step 1 → template fetch is SKIPPED entirely (the `if (!agentSoulMd || ...)` guard prevents it) → profile fields unchanged
3. **Restart after agent self-edits**: Agent modified its SOUL.md during a session → on restart, the modified content is preserved, template is NOT re-applied
4. **First boot with TEMPLATE_ID but partial profile**: Agent has soulMd set but not claudeMd → only missing fields get template content, existing fields are preserved
5. **TEMPLATE_ID set but registry unreachable**: Falls back to `generateDefault*()` — does NOT error or block boot

**Implementation approach**: Mock the HTTP fetch to the registry, use an in-memory SQLite DB. The key assertion is: after a profile is saved to the DB, subsequent `GET /me` returns those fields, and the `if (!agentSoulMd || ...)` check skips template fetching entirely.

This test runs as part of `bun test` in CI.
